## 1. 오늘 배운 전체 흐름

오늘 수업은 **IT 기초 → 웹 서비스 구조 → 개발 도구 → 데이터 저장 → API → 배포/운영 → 블로그와 SEO** 순서로 연결된다.

가장 큰 흐름은 아래와 같다.

```text
컴퓨터는 0과 1로 동작한다
→ 사람이 쓰기 편하게 프로그래밍 언어가 등장했다
→ 웹은 클라이언트와 서버가 요청/응답으로 통신하는 구조다
→ 웹 서비스는 Front-end, Back-end, Database, API로 구성된다
→ 개발 생산성을 높이기 위해 라이브러리와 프레임워크를 사용한다
→ Git/GitHub로 코드와 학습 자료를 관리한다
→ 서비스가 운영되면 데이터가 쌓이고, DB/API/분석이 중요해진다
→ 블로그를 만들고 SEO, Google Search Console, Google Analytics로 노출과 방문 데이터를 관리한다
```

내 수업 메모에서 특히 중요하게 보이는 방향은 다음과 같다.

- RDB, NoSQL, VectorDB를 모두 공부한다.
- API 문서를 읽고 활용할 수 있어야 한다.
- API를 활용하는 것뿐만 아니라, 다른 사람이 활용할 수 있도록 제공하는 관점도 중요하다.
- 데이터 분석가도 SQL, API, 웹 구조, Git, 블로그 운영을 알아야 한다.
- 실무에서는 단순 문법보다 속도, 과부하, 운영 안정성이 중요하다.
- GitHub 블로그를 만들고 Google Search Console, Google Analytics까지 연결해야 한다.
- Linux, 서버, 배포 개념은 웹 서비스 운영과 연결된다.
- 질적 연구와 양적 연구, 보고서 작성 능력도 데이터 분석가에게 중요하다.

## 2. 핵심 개념 정리

---

### 2.1 컴퓨터 기본 개념

- 한 줄 정의:  
  컴퓨터는 0과 1의 신호를 기반으로 정보를 처리하는 기계다.

- 쉽게 설명:  
  사람은 글자, 숫자, 그림으로 생각하지만 컴퓨터는 결국 전기 신호의 켜짐/꺼짐처럼 단순한 상태를 조합해서 정보를 처리한다.

- 왜 중요한지:  
  프로그래밍 언어, 운영체제, 서버, 데이터베이스 같은 모든 IT 개념의 바탕이 된다.  
  High Level과 Low Level을 구분하는 기준도 여기서 출발한다.

- 예시:  
  TV On/Off, 컴퓨터 On/Off, 에어컨 On/Off처럼 생활 속 기계도 1/0 상태로 이해할 수 있다.  
  컴퓨터 내부에서는 이런 0과 1이 훨씬 복잡하게 조합된다.

- 헷갈릴 수 있는 점:  
  “파이썬을 배운다”는 것은 컴퓨터가 직접 이해하는 언어를 쓰는 것이 아니다.  
  사람이 쓰기 쉬운 언어를 사용하고, 이를 컴퓨터가 이해할 수 있는 형태로 바꾸는 과정이다.

- 내 말로 다시 정리:  
  컴퓨터는 결국 0과 1밖에 모르고, 개발자는 사람이 이해하기 쉬운 언어로 명령을 작성한 뒤 컴퓨터가 알아들을 수 있게 전달하는 사람이다.

---

### 2.2 프로그래밍 언어 / High Level / Low Level

- 한 줄 정의:  
  프로그래밍 언어는 사람이 컴퓨터에게 명령을 내리기 위해 사용하는 언어다.

- 쉽게 설명:  
  C, C++, Java, Python, JavaScript 같은 언어는 모두 컴퓨터에게 일을 시키는 도구다.  
  다만 기계에 가까운 언어일수록 Low Level, 사람의 사고방식에 가까울수록 High Level에 가깝다.

- 왜 중요한지:  
  데이터 분석을 할 때는 Python이 편하지만, 고성능 시스템, 임베디드, 현장 적용 시스템에서는 C/C++ 같은 언어가 중요할 수 있다.

- 예시:

```text
Python → 데이터 분석, 자동화, 머신러닝, 빠른 실험
C/C++ → 성능이 중요한 시스템, 임베디드, 하드웨어 제어
JavaScript → 웹 프론트엔드, 웹 서비스
Java/Kotlin/Swift → 앱 또는 백엔드 개발
```

- 헷갈릴 수 있는 점:  
  Python이 느리다는 말이 “Python은 쓸모없다”는 뜻은 아니다.  
  목적에 따라 적합한 도구가 다를 뿐이다.

- 내 말로 다시 정리:  
  분석은 Python으로 시작해도 되지만, 실제 서비스나 차량/임베디드 현장까지 생각하면 언어별 강점과 한계를 알아야 한다.

---

### 2.3 운영체제

- 한 줄 정의:  
  운영체제는 컴퓨터라는 기계를 사람이 쉽게 다룰 수 있도록 도와주는 시스템 소프트웨어다.

- 쉽게 설명:  
  우리가 컴퓨터나 스마트폰을 사용할 때 직접 하드웨어를 제어하지 않아도 되는 이유는 운영체제가 중간에서 관리해주기 때문이다.

- 왜 중요한지:  
  개발 환경, 서버 운영, 배포, 앱 실행 방식이 운영체제에 따라 달라진다.

- 예시:

```text
Windows
macOS
Linux
Android
iOS
```

- 헷갈릴 수 있는 점:  
  운영체제는 단순히 화면 디자인이 아니다.  
  프로그램 실행, 파일 관리, 메모리 관리, 하드웨어 제어 등 핵심 역할을 한다.

- 내 말로 다시 정리:  
  운영체제는 컴퓨터를 사람이 쓸 수 있게 만들어주는 기본 시스템이다.

### 2.4 웹과 웹 서비스

- 한 줄 정의:  
  웹 서비스는 사용자의 요청을 서버가 처리하고, 그 결과를 다시 사용자에게 응답하는 구조다.

- 쉽게 설명:  
  사용자가 웹사이트에 접속하거나 버튼을 누르면 클라이언트가 서버에 요청을 보낸다.  
  서버는 필요한 로직을 처리하거나 DB에서 데이터를 가져와 다시 응답한다.

- 왜 중요한지:  
  데이터 분석가도 웹 구조를 모르면 데이터가 어디서 생기고, 어디에 저장되고, 어떻게 API로 제공되는지 이해하기 어렵다.

- 예시:

```text
사용자 로그인
→ 클라이언트가 서버에 로그인 요청
→ 서버가 DB에서 회원 정보 확인
→ 로그인 성공/실패 응답
→ 클라이언트가 결과 화면 표시
```

- 헷갈릴 수 있는 점:  
  웹사이트 화면만 보는 것이 웹 서비스의 전부가 아니다.  
  화면 뒤에는 서버, 백엔드 로직, 데이터베이스, API가 있다.

- 내 말로 다시 정리:  
  웹 서비스는 “화면”이 아니라 “요청 → 처리 → 저장 → 응답”의 전체 흐름이다.

---

### 2.5 클라이언트와 서버

- 한 줄 정의:  
  클라이언트는 요청하는 쪽이고, 서버는 요청을 처리해 응답하는 쪽이다.

- 쉽게 설명:  
  내가 스마트폰 앱에서 버튼을 누르면 앱이 클라이언트가 되고, 요청을 받은 서버가 처리 결과를 돌려준다.

- 왜 중요한지:  
  웹, 앱, API, 데이터베이스, 배포를 이해하는 기본 구조다.

- 예시:

```text
배달앱 사용자 → 주문 요청
배달앱 서버 → 주문 처리
치킨집 → 주문 확인
사용자 앱 → 주문 완료 화면 표시
```

- 헷갈릴 수 있는 점:  
  서버는 단순히 컴퓨터 한 대만 의미하지 않는다.  
  요청을 처리하는 역할 전체를 의미할 수 있다.

- 내 말로 다시 정리:  
  클라이언트와 서버는 웹 서비스의 가장 기본적인 대화 구조다.

---

### 2.6 HTTP / Request / Response

- 한 줄 정의:  
  HTTP는 웹에서 클라이언트와 서버가 약속된 방식으로 데이터를 주고받기 위한 통신 규칙이다.

- 쉽게 설명:  
  사람이 대화할 때 언어와 규칙이 필요하듯, 컴퓨터끼리도 데이터를 주고받기 위한 약속이 필요하다.  
  웹에서는 그 대표적인 약속이 HTTP다.

- 왜 중요한지:  
  API, 웹 서버, 백엔드, 프론트엔드, 크롤링을 이해하려면 요청과 응답 구조를 알아야 한다.

- 예시:

```text
Request:
사용자가 상품 페이지를 열어달라고 요청

Response:
서버가 상품 정보와 화면 데이터를 응답
```

- 헷갈릴 수 있는 점:  
  HTTP는 웹페이지 주소 자체가 아니라, 웹에서 데이터를 주고받는 방식이다.

- 내 말로 다시 정리:  
  웹에서 어떤 행동을 하면 거의 항상 요청이 발생하고, 서버는 그에 대한 응답을 준다.

---

### 2.7 Front-end

- 한 줄 정의:  
  Front-end는 사용자가 직접 보고 조작하는 화면 영역이다.

- 쉽게 설명:  
  로그인 화면, 버튼, 입력창, 메뉴, 반응형 화면처럼 사용자가 처음 만나는 부분이다.

- 왜 중요한지:  
  사용자는 서비스의 내부 구조를 보지 못한다.  
  결국 사용 경험은 프론트엔드에서 크게 결정된다.

- 예시:

```text
로그인 화면
회원가입 화면
상품 목록 화면
검색창
버튼
지도 화면
반응형 웹 화면
```

- 헷갈릴 수 있는 점:  
  프론트엔드는 단순히 “예쁘게 꾸미는 일”만이 아니다.  
  서버와 통신하고, 사용자 행동을 수집하고, UX 흐름을 설계하는 역할도 포함된다.

- 내 말로 다시 정리:  
  프론트엔드는 사용자가 서비스를 실제로 경험하는 입구다.

---

### 2.8 UI / UX

- 한 줄 정의:  
  UI는 사용자가 보는 화면 요소이고, UX는 사용자가 서비스를 이용하며 느끼는 전체 경험이다.

- 쉽게 설명:  
  UI는 버튼, 입력창, 메뉴, 색상 같은 화면 구성 요소다.  
  UX는 사용자가 목적을 달성하기까지 편리했는지, 헷갈리지 않았는지, 불편하지 않았는지를 포함한다.

- 왜 중요한지:  
  기능이 좋아도 사용자가 사용하기 어렵다면 서비스 품질은 낮아진다.

- 예시:

```text
UI:
로그인 버튼, 아이디 입력창, 비밀번호 입력창

UX:
사용자가 로그인 과정을 어렵지 않게 완료하는 흐름
```

- 헷갈릴 수 있는 점:  
  UI가 예쁘다고 UX가 무조건 좋은 것은 아니다.  
  사용자가 원하는 일을 쉽게 끝낼 수 있어야 UX가 좋다.

- 내 말로 다시 정리:  
  UI는 화면이고, UX는 그 화면을 쓰는 전체 경험이다.

---

### 2.9 Back-end / Web Server / Database

- 한 줄 정의:  
  Back-end는 화면 뒤에서 요청을 처리하고 데이터를 저장·조회하는 서버 영역이다.

- 쉽게 설명:  
  사용자가 로그인 버튼을 누르면 화면에서는 버튼만 보이지만, 뒤에서는 서버가 계정 정보를 확인하고 DB에서 데이터를 조회한다.

- 왜 중요한지:  
  서비스의 핵심 로직, 데이터 처리, 보안, 속도, 안정성이 백엔드와 DB에서 결정된다.

- 예시:

```text
클라이언트
→ Web Server
→ Back-end 로직
→ Database
→ Back-end 처리 결과
→ Web Server
→ 클라이언트 응답
```

- 헷갈릴 수 있는 점:  
  Web Server와 Back-end가 항상 같은 의미는 아니다.  
  Web Server는 HTTP 요청을 받아 전달하는 역할을 하고, Back-end는 실제 비즈니스 로직을 처리하는 영역으로 나눠 볼 수 있다.

- 내 말로 다시 정리:  
  백엔드는 화면 뒤에서 실제 일을 처리하는 부분이고, DB는 그 과정에서 필요한 정보를 저장하는 곳이다.

---

### 2.10 Database / RDB / NoSQL / VectorDB

- 한 줄 정의:  
  데이터베이스는 데이터를 저장하고 필요할 때 꺼내 쓰기 위한 시스템이다.

- 쉽게 설명:  
  서비스가 커지면 회원 정보, 결제 정보, 게시글, 이미지, 로그 같은 데이터가 계속 쌓인다.  
  이 데이터를 목적에 맞게 저장하는 공간이 DB다.

- 왜 중요한지:  
  서비스가 성장하면 데이터 저장, 조회 속도, 분석 가능성이 매우 중요해진다.  
  데이터 분석가도 DB를 모르면 실무 데이터에 접근하고 활용하기 어렵다.

- 예시:

```text
RDB:
- MySQL
- Oracle
- PostgreSQL
- 정형 데이터
- 웹 개발, 데이터 분석에 자주 사용

NoSQL:
- MongoDB
- Elasticsearch
- 이미지, 자연어, 로그, 비정형 데이터 등에 사용

VectorDB:
- ChromaDB 등
- LLM 관련 프로젝트, 임베딩 검색, 유사도 검색에 활용
```

- 헷갈릴 수 있는 점:  
  DB 공부는 단순히 SQL 문법만 외우는 것이 아니다.  
  쿼리 속도, 지연, 과부하, 인덱스, 데이터 구조까지 생각해야 한다.

- 내 말로 다시 정리:  
  데이터 분석가도 DB를 알아야 한다. 데이터는 결국 어디엔가 저장되고, 잘 꺼내 써야 분석도 가능하다.

---

### 2.11 SQL 학습 범위

- 한 줄 정의:  
  SQL은 RDB에서 데이터를 조회, 생성, 수정, 삭제하기 위한 언어다.

- 쉽게 설명:  
  엑셀에서 필터를 걸고 원하는 데이터를 찾듯이, SQL은 데이터베이스에서 원하는 데이터를 가져오기 위한 언어다.

- 왜 중요한지:  
  데이터 분석가와 개발자 모두 실무에서 데이터를 다루려면 SQL을 알아야 한다.

- 예시 학습 범위:

```text
기본 문법
SELECT
WHERE
GROUP BY
ORDER BY
JOIN
서브쿼리
윈도우 함수
PL/SQL
성능 튜닝 기초
```

- 헷갈릴 수 있는 점:  
  SQL은 문법을 아는 것에서 끝나지 않는다.  
  실무에서는 쿼리가 느리지 않은지, DB에 부담을 주지 않는지가 중요하다.

- 내 말로 다시 정리:  
  SQL은 데이터 분석가의 기본기이고, 실무에서는 성능까지 봐야 한다.

---

### 2.12 API

- 한 줄 정의:  
  API는 다른 프로그램이나 사람이 특정 기능이나 데이터를 사용할 수 있게 열어둔 연결 통로다.

- 쉽게 설명:  
  내가 만든 서비스의 데이터를 다른 사람이 정해진 방식으로 가져가거나 사용할 수 있게 해주는 문이다.

- 왜 중요한지:  
  API 활용은 기본이고, 다른 사람이 내 API를 활용할 수 있게 만드는 능력도 중요하다.  
  데이터를 API로 제공하면 무료 지원, 유료화, 플랫폼 비즈니스로 연결될 수 있다.

- 예시:

```text
날씨 API:
특정 지역의 날씨 데이터를 제공

결제 API:
카카오페이, 네이버페이 같은 결제 기능 연결

배송 API:
배송 상태 조회 기능 제공

차량 데이터 API:
차량 운행 정보, 센서 데이터, 진단 데이터 제공 가능
```

- 헷갈릴 수 있는 점:  
  API는 단순히 “주소”가 아니다.  
  어떤 요청을 보내면 어떤 응답을 주는지에 대한 규칙과 문서까지 포함해서 이해해야 한다.

- 내 말로 다시 정리:  
  API는 서비스를 외부와 연결하는 공식 통로이고, 데이터 사업화와도 연결된다.

---

### 2.13 REST API / CRUD / GET·POST·PUT·DELETE

- 한 줄 정의:  
  REST API는 웹에서 자원을 일정한 규칙으로 다루기 위한 API 설계 방식이고, CRUD는 데이터 생성·조회·수정·삭제의 기본 동작이다.

- 쉽게 설명:  
  게시글을 예로 들면 글 작성은 Create, 글 조회는 Read, 글 수정은 Update, 글 삭제는 Delete다.

- 왜 중요한지:  
  서비스 기획자나 데이터 분석가도 API 문서를 읽으려면 최소한 GET, POST, PUT, DELETE 개념은 알아야 한다.

- 예시:

```text
Create → POST → 새 게시글 작성
Read   → GET → 게시글 조회
Update → PUT → 게시글 수정
Delete → DELETE → 게시글 삭제
```

- 헷갈릴 수 있는 점:  
  실제 API에서는 PUT 대신 PATCH를 쓰는 경우도 있다.  
  해당 수업자료에서 PATCH까지 다루었는지는 추가 확인 필요.

- 내 말로 다시 정리:  
  CRUD는 거의 모든 서비스 데이터 처리의 기본 동작이다.

---

### 2.14 동기 / 비동기

- 한 줄 정의:  
  동기는 하나가 끝날 때까지 기다리는 방식이고, 비동기는 기다리는 동안 다른 일을 처리할 수 있는 방식이다.

- 쉽게 설명:  
  동기는 카페에서 주문하고 음료가 나올 때까지 줄 앞에서 기다리는 느낌이다.  
  비동기는 진동벨을 받고 자리에서 다른 일을 하다가 알림이 오면 받으러 가는 느낌이다.

- 왜 중요한지:  
  웹 서비스에서 서버 응답, 데이터 처리, 사용자 경험에 영향을 준다.

- 예시:

```text
동기:
로그인 요청 후 결과가 올 때까지 기다림

비동기:
파일 업로드를 진행하면서 다른 화면 조작 가능
```

- 헷갈릴 수 있는 점:  
  비동기가 항상 좋은 것은 아니다.  
  처리 순서가 중요하거나 즉시 결과 확인이 필요한 경우 동기가 더 적합할 수 있다.

- 내 말로 다시 정리:  
  동기와 비동기는 좋고 나쁨이 아니라 상황에 맞는 처리 방식이다.

---

### 2.15 라이브러리

- 한 줄 정의:  
  라이브러리는 자주 쓰는 기능을 모아둔 코드 도구다.

- 쉽게 설명:  
  매번 모든 코드를 처음부터 만들지 않고, 이미 검증된 기능을 가져와서 쓰는 것이다.

- 왜 중요한지:  
  개발 생산성과 효율성을 높인다.  
  반복되는 코드를 줄이고, 검증된 기능을 활용할 수 있다.

- 예시:

```text
Pandas:
데이터 분석 및 가공

Chart.js:
차트 시각화

Passport.js:
로그인/회원가입 인증 구현 보조
```

- 헷갈릴 수 있는 점:  
  라이브러리를 많이 쓴다고 무조건 좋은 것은 아니다.  
  부적절하게 사용하면 성능 저하나 유지보수 문제가 생길 수 있다.

- 내 말로 다시 정리:  
  라이브러리는 검증된 부품을 가져다 쓰는 것이지만, 왜 쓰는지 알고 써야 한다.

---

### 2.16 프레임워크

- 한 줄 정의:  
  프레임워크는 프로그램을 효율적으로 만들기 위해 기본 구조와 규칙을 제공하는 틀이다.

- 쉽게 설명:  
  빵틀처럼 정해진 구조 안에서 코드를 작성하게 해주는 도구다.

- 왜 중요한지:  
  협업과 유지보수에 유리하다.  
  공통 규칙을 제공하므로 코드 구조를 맞추기 쉽다.

- 예시:

```text
Django:
Python 기반 웹 개발 프레임워크

추가 프레임워크 예시:
수업자료에서 충분히 확인되지 않음
→ 추가 확인 필요
```

- 헷갈릴 수 있는 점:  
  라이브러리는 내가 필요할 때 가져다 쓰는 느낌이고, 프레임워크는 정해진 구조 안에서 내가 코드를 채워 넣는 느낌에 가깝다.

- 내 말로 다시 정리:  
  라이브러리는 도구 상자, 프레임워크는 작업 틀이다.

---

### 2.17 Git / GitHub / Version Control

- 한 줄 정의:  
  Git은 코드와 문서의 변경 이력을 관리하는 버전 관리 도구이고, GitHub는 이를 온라인에서 저장·공유하는 서비스다.

- 쉽게 설명:  
  작업하다가 문제가 생겼을 때 이전 상태로 돌아갈 수 있고, 누가 어떤 내용을 수정했는지도 확인할 수 있다.

- 왜 중요한지:  
  개인 학습 기록, 프로젝트 코드, 블로그 소스, 협업 작업을 관리하는 기본 도구다.

- 예시:

```text
Git:
내 컴퓨터에서 변경 이력을 관리하는 도구

GitHub:
Git 저장소를 온라인에 올리고 공유하는 서비스

GitHub Pages:
GitHub 저장소를 이용해 정적 웹사이트를 배포하는 기능
```

- 헷갈릴 수 있는 점:  
  Git과 GitHub는 같은 것이 아니다.  
  Git은 버전 관리 도구이고, GitHub는 Git 저장소를 온라인에서 관리하는 서비스다.

- 내 말로 다시 정리:  
  Git은 내 작업의 시간 기록이고, GitHub는 그 기록을 보관하고 보여주는 공간이다.

---

### 2.18 GitHub 블로그 / Hugo

- 한 줄 정의:  
  GitHub 블로그는 GitHub Pages를 이용해 개인 블로그를 운영하는 방식이고, Hugo는 정적 사이트 생성 도구다.

- 쉽게 설명:  
  Markdown으로 글을 작성하고, Hugo가 이를 웹사이트 형태로 변환해준다.  
  GitHub Pages를 통해 인터넷에 공개할 수 있다.

- 왜 중요한지:  
  학습 기록, 프로젝트 정리, 포트폴리오, 검색 노출까지 연결할 수 있다.

- 예시 작업 흐름:

```text
1. GitHub 회원가입
2. Git 설치
3. Hugo 설치
4. 계정ID.github.io 저장소 생성
5. 블로그 소스 저장소 생성
6. 테마 적용
7. Git submodule 개념 이해
8. 글 작성
9. GitHub에 업로드
10. Google Search Console / Analytics 연결
```

- 헷갈릴 수 있는 점:  
  GitHub Pages 저장소와 블로그 소스 저장소를 분리해서 운영할 수 있다.  
  이때 submodule 개념이 필요할 수 있다.

- 내 말로 다시 정리:  
  GitHub 블로그는 공부 기록을 온라인 포트폴리오로 쌓기 위한 기반이다.

---

### 2.19 오픈소스

- 한 줄 정의:  
  오픈소스는 코드가 공개되어 누구나 확인하고 활용할 수 있는 소프트웨어다.

- 쉽게 설명:  
  여러 사람이 함께 코드를 개선하고 공유하는 방식이다.

- 왜 중요한지:  
  많은 개발 도구와 라이브러리, 프레임워크가 오픈소스 기반이다.  
  다만 무료라고 해서 아무 규칙 없이 쓰는 것은 아니고 라이선스를 확인해야 한다.

- 예시:

```text
Linux
MySQL
WordPress
Apache
일부 JavaScript 라이브러리
일부 Python 패키지
```

- 헷갈릴 수 있는 점:  
  오픈소스 = 무조건 무료·무제한 사용이 아니다.  
  라이선스 조건을 지켜야 한다.

- 내 말로 다시 정리:  
  오픈소스는 공개된 코드 자산이지만, 사용 규칙을 지켜야 하는 자산이다.

---

### 2.20 클라우드

- 한 줄 정의:  
  클라우드는 인터넷을 통해 서버와 컴퓨팅 자원을 필요한 만큼 빌려 쓰는 방식이다.

- 쉽게 설명:  
  직접 서버 컴퓨터를 사서 운영하지 않고, 필요한 만큼 서버를 빌려 쓰고 사용량에 따라 비용을 내는 방식이다.

- 왜 중요한지:  
  서비스 운영에서는 트래픽 증가와 감소에 대응해야 한다.  
  클라우드는 서버 증설과 폐기를 유연하게 할 수 있다.

- 예시:

```text
AWS
Google Cloud Platform
Microsoft Azure

구체적인 수업자료 내 클라우드 서비스 목록은 추가 확인 필요
```

- 헷갈릴 수 있는 점:  
  클라우드를 쓴다고 비용 문제가 사라지는 것은 아니다.  
  오히려 잘못 설정하면 예상보다 많은 비용이 나올 수 있다.

- 내 말로 다시 정리:  
  클라우드는 서버를 직접 사지 않고 필요한 만큼 빌려 쓰는 방식이다.

---

### 2.21 SaaS

- 한 줄 정의:  
  SaaS는 Software as a Service의 약자로, 설치 없이 인터넷으로 사용하는 소프트웨어 서비스다.

- 쉽게 설명:  
  내 컴퓨터에 프로그램을 직접 설치하지 않아도 웹에서 바로 사용할 수 있는 서비스다.

- 왜 중요한지:  
  협업이 쉽고, 설치 과정 없이 바로 사용할 수 있으며, 클라우드 기반 서비스의 대표적인 형태다.

- 예시:

```text
Figma
Notion
Google Docs
Slack

수업자료에서 언급된 구체 서비스 목록은 추가 확인 필요
```

- 헷갈릴 수 있는 점:  
  클라우드는 서버/인프라 관점이고, SaaS는 사용자가 바로 쓰는 소프트웨어 서비스 관점이다.

- 내 말로 다시 정리:  
  클라우드는 빌려 쓰는 서버 환경이고, SaaS는 설치 없이 바로 쓰는 서비스다.

---

### 2.22 SEO / 검색엔진 최적화

- 한 줄 정의:  
  SEO는 내가 만든 웹페이지가 검색엔진에 잘 발견되고 노출되도록 최적화하는 작업이다.

- 쉽게 설명:  
  블로그 글을 써도 검색엔진이 찾지 못하면 방문자가 오기 어렵다.  
  제목, 키워드, 구조, 색인, 사이트 등록 등이 중요하다.

- 왜 중요한지:  
  GitHub 블로그를 만들고 끝나는 것이 아니라, Google Search Console과 Google Analytics를 연결해 검색 노출과 방문 데이터를 확인해야 한다.

- 예시:

```text
제목에 핵심 키워드 포함
본문 구조를 명확하게 작성
검색엔진에 사이트 등록
사이트맵 제출
Google Search Console 연결
Google Analytics 연결
```

- 헷갈릴 수 있는 점:  
  SEO는 단순히 해시태그를 많이 붙이는 것이 아니다.  
  검색엔진이 페이지를 크롤링하고 색인할 수 있게 구조화하는 작업이 중요하다.

- 내 말로 다시 정리:  
  SEO는 내가 쓴 글이 검색엔진에 잘 읽히도록 정리하는 작업이다.

---

### 2.23 크롤링

- 한 줄 정의:  
  크롤링은 컴퓨터가 웹페이지를 자동으로 방문해 정보를 수집하는 작업이다.

- 쉽게 설명:  
  사람이 웹페이지를 하나씩 열어보는 대신, 프로그램이 웹페이지를 돌아다니며 필요한 데이터를 가져오는 것이다.

- 왜 중요한지:  
  검색엔진도 웹을 크롤링해 정보를 수집하고, 데이터 분석에서도 웹 데이터를 수집할 때 크롤링을 활용할 수 있다.

- 예시:

```text
검색엔진 봇이 웹페이지를 방문
→ 페이지 내용을 수집
→ 색인 생성
→ 검색결과에 노출
```

- 헷갈릴 수 있는 점:  
  크롤링이 항상 자유롭게 가능한 것은 아니다.  
  사이트 이용약관, robots.txt, 저작권, 개인정보 이슈는 별도 확인이 필요하다.  
  국내 기준과 실무 적용 범위는 추가 확인 필요.

- 내 말로 다시 정리:  
  크롤링은 웹 데이터를 자동으로 모으는 기술이고, SEO와 데이터 분석 양쪽에 연결된다.

## 7. 용어 사전

| 용어 | 의미 |
|---|---|
| 컴퓨터 | 0과 1을 기반으로 정보를 처리하는 기계 |
| 프로그래밍 언어 | 사람이 컴퓨터에게 명령하기 위해 사용하는 언어 |
| 운영체제 | 컴퓨터를 쉽게 다룰 수 있게 해주는 시스템 소프트웨어 |
| 클라이언트 | 서버에 요청을 보내는 쪽 |
| 서버 | 요청을 처리하고 응답하는 쪽 |
| HTTP | 웹에서 데이터를 주고받기 위한 통신 규칙 |
| Request | 클라이언트가 서버에 보내는 요청 |
| Response | 서버가 클라이언트에게 돌려주는 응답 |
| Front-end | 사용자가 직접 보는 화면 영역 |
| Back-end | 화면 뒤에서 로직과 데이터를 처리하는 영역 |
| Database | 데이터를 저장하고 관리하는 시스템 |
| RDB | 정형 데이터를 테이블 형태로 저장하는 DB |
| NoSQL | 비정형 또는 유연한 구조의 데이터를 저장하는 DB |
| VectorDB | 벡터 임베딩을 저장하고 유사도 검색에 활용하는 DB |
| API | 기능이나 데이터를 외부에서 사용할 수 있게 하는 연결 통로 |
| CRUD | 데이터 생성, 조회, 수정, 삭제 |
| GET | 데이터를 조회할 때 주로 사용하는 HTTP 메서드 |
| POST | 데이터를 생성할 때 주로 사용하는 HTTP 메서드 |
| PUT | 데이터를 수정할 때 주로 사용하는 HTTP 메서드 |
| DELETE | 데이터를 삭제할 때 주로 사용하는 HTTP 메서드 |
| 라이브러리 | 자주 사용하는 기능을 모아둔 코드 도구 |
| 프레임워크 | 개발을 위한 기본 구조와 규칙을 제공하는 틀 |
| Git | 변경 이력을 관리하는 버전 관리 도구 |
| GitHub | Git 저장소를 온라인에서 관리하는 서비스 |
| 오픈소스 | 코드가 공개되어 활용 가능한 소프트웨어 |
| 클라우드 | 인터넷으로 서버와 컴퓨팅 자원을 빌려 쓰는 방식 |
| SaaS | 설치 없이 인터넷으로 사용하는 소프트웨어 서비스 |
| SEO | 검색엔진에 잘 노출되도록 웹페이지를 최적화하는 작업 |
| 크롤링 | 웹페이지를 자동으로 방문해 정보를 수집하는 작업 |
| 색인 | 검색엔진이 수집한 정보를 검색 가능하게 정리하는 과정 |
